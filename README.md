# 3D Portfolio Starter (Three.js + React Three Fiber)

This project is a small 3D engine built with React Three Fiber: a click-to-move player on a flat plane, in-scene trigger zones, and a minimal global store (Zustand) that records section activation. You removed the built-in modal overlay — the engine still updates `activeSection` in state so you can consume it with your own UI.

## What changed
- The modal overlay UI was removed from the app (no automatic HTML overlay shown).
- Modal-related CSS was removed from `src/styles.css`.
- `src/App.css` is currently unused and safe to delete if you don't need it.
- Engine pieces remain: player movement, triggers, world, and store.

## Interaction Flow (engine)

1. User clicks the ground (`src/scene/Ground.tsx`).
2. Ray hit point becomes `targetPosition` in the store.
3. `Player` moves toward `targetPosition` every frame (`src/scene/Player.tsx`).
4. Entering a trigger zone updates `activeSection` in the store (`projects`, `about`, `contact`).
5. There is no built-in overlay shown by default — the app only updates `activeSection` in state. Build a UI that reads `useWorldStore` to present content.

## Project file structure (what each file/folder does)

- `index.html` — Vite HTML entry.
- `package.json` — scripts and dependencies.
- `vite.config.ts` — Vite configuration (React plugin).
- `eslint.config.js` — ESLint flat config used by the project.
- `tsconfig.*.json` — TypeScript configurations.
- `public/` — static files copied to the build output (favicons, static models, etc.).

src/
- `main.tsx` — app bootstrap; imports global styles and mounts `App`.
- `App.tsx` — top-level React component; composes the `WorldCanvas` and (previously) overlay UI.
- `styles.css` — global styles used by the app (includes game-shell styling and modal styles removed).
- `App.css` — unused CSS file (was previously used for landing/modal visuals).
- `assets/` — small static images used in the UI (if any).
- `scene/` — 3D scene components:
  - `WorldCanvas.tsx` — creates the `<Canvas>` and registers lights, environment, and scene children.
  - `Ground.tsx` — renders the visible ground and an invisible click plane that sets movement targets.
  - `Player.tsx` — player mesh and movement logic (frame loop, facing interpolation, position updates).
  - `Triggers.tsx` — trigger definitions, in-scene markers, and `useTriggerDefs()` helper.
- `store/`:
  - `worldStore.ts` — Zustand store: `playerPosition`, `targetPosition`, `playerSpeed`, `activeSection`, cooldown logic, and actions (`setTarget`, `setPlayerPosition`, `openSection`, etc.).
- `types/`:
  - `world.ts` — shared types for `SectionId` and `TriggerDef`.
- `ui/`:
  - `SectionsOverlay.tsx` — (removed/optional) previously provided the modal overlay that read `activeSection` and rendered HTML UI. If you restore it, import it in `App.tsx`.

- `dist/` — build output (generated by `npm run build`; typically listed in `.gitignore`).

## How to consume section state now

The engine still sets `activeSection` in `src/store/worldStore.ts`. To show UI when a trigger is entered, create a React component that reads `useWorldStore((s) => s.activeSection)` and renders whatever you want (panel, sidebar, toast, etc.).

Example consumer sketch:

```tsx
import { useWorldStore } from './src/store/worldStore'

function SectionPanel(){
  const active = useWorldStore((s) => s.activeSection)
  if(active === 'none') return null
  return <div className="section-panel">Active: {active}</div>
}
```

## Debug checklist (short)

- If movement does not work: verify `Ground` pointer events and that `setTarget` runs.
- If triggers don't match expectations: tune `Triggers` centers and radii.
- If player jitters: increase `arriveThreshold` in `Player.tsx`.
